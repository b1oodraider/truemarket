## 1. Роль Claude Code в этом проекте

Claude Code работает как **старший инженер-архитектор и технический лид**.

### Что Claude делает без напоминаний
- Принимает и **обосновывает** технические решения через ADR
- Пишет **production-grade** код (не proof-of-concept)
- **Проактивно** выявляет проблемы до их возникновения
- Соблюдает все договорённости из этого документа

### Что Claude не делает
- Не просит разрешения на очевидные технические решения
- Не задаёт вопросы, ответ на которые есть в контексте
- Не пишет код без тестов для бизнес-логики
- Не коммитит секреты, не использует магические числа, не игнорирует ошибки

### Что Claude делает обязательно
- **Сообщает о блокерах явно** в формате из раздела 11
- **Останавливается** при выполнении любого стоп-условия (раздел 12)
- **Создаёт ADR** перед нетривиальным решением, не после

## 2. Ключевые дифференциаторы продукта

Эти **шесть** пунктов — **бизнес-фундамент**. Любое техническое решение, противоречащее им, отклоняется.

| # | Дифференциатор | Что это значит для кода |
|---|---|---|
| 1 | **Доверие** | Верифицированные отзывы (только после `delivered`), защита от накрутки рейтингов |
| 2 | **Честное соответствие** | Верификация совпадения фото с реальным товаром (выборочная + краудсорсинг) |
| 3 | **Честные алгоритмы** | Органическая выдача только по релевантности; реклама помечается, никогда не смешивается с органикой |
| 4 | **Прозрачные условия** | Понятные комиссии, предсказуемая система, прямая аналитика для продавцов |
| 5 | **Единая оплата** | Покупатель привязывает карту один раз, платит на всём маркетплейсе. Реализуется через агентскую модель платформы (ADR-008): платформа — платёжный агент продавцов ИП/ООО; ЮKassa проводит все платежи и выпускает агентские чеки |
| 6 | **Трекинг доставки** | Прямые интеграции с российскими курьерами (СДЭК, Почта России, Boxberry, Яндекс.Доставка) — см. ADR-005. AfterShip/TrackingMore отклонены из-за рисков 152-ФЗ |

## 3. Экономическая модель

### Базовая комиссия по категориям
| Категория | Базовая комиссия |
|---|---|
| Одежда / текстиль | 12% |
| Ювелирные изделия | 10% |
| Товары для дома | 10% |

### Бонусы снижения комиссии (суммируются, **максимум −6%**)
| Бонус | Размер |
|---|---|
| Российское производство | −2% |
| Сертификация (ГОСТ / ТР ТС) | −1% |
| Быстрая доставка (уровень Trusted+) | −1.5% |
| Верифицированный продавец | −1.5% |

### ✅ Решено в [ADR-004](docs/adr/ADR-004-commission-bonus-stacking.md)
Бонус «Быстрая доставка (Trusted+)» из списка выше и автоматическая скидка уровня `Trusted Seller` — **один и тот же бонус, не складываются**. Технически вычисляется через достижение уровня `Trusted` или выше. Уровни `Verified Local / Trusted / National / All-Russia` взаимно исключающие. Дополнительные бонусы (российское производство, сертификация, верификация) складываются между собой и с уровневой скидкой, **с потолком −6%**.

### Минимальная итоговая комиссия
- Одежда / текстиль: 12% − 6% = **6%**
- Ювелирка / дом: 10% − 6% = **4%**

### Уровни продавца по географии доставки
| Уровень | Условие | Бенефит |
|---|---|---|
| `Verified Local` | Доставка по региону | Базовая комиссия |
| `Trusted Seller` | ЕЧР¹ ≤ 7 дней | −1.5% + badge |
| `National` | ЕЧР + Урал + Сибирь ≤ 10 дней | −3% + приоритет в региональной выдаче |
| `All-Russia` | Все округа включая ДВ² | −4.5% + featured статус |

¹ **ЕЧР** = Европейская часть России (от Калининграда до Урала включительно).
² **ДВ** = Дальневосточный федеральный округ.

### Дополнительная монетизация
- Явная реклама (обязательная пометка `Реклама`, **не влияет на органику**)
- Платные инструменты аналитики (опционально для продавцов)

### Постпродажный опыт
- Возвраты и гарантии — **на стороне продавцов**
- Платформа: диспетчер споров + агрегатор трекинга

## 4. Технологический стек

## 5. Архитектура

### 5.1. Стиль архитектуры
**Модульный монолит на старте → миграция к микросервисам по мере роста.**

Обоснование (фиксируется в ADR-006): скорость разработки MVP важнее преждевременной оптимизации. Границы модулей **должны совпадать** с будущими границами сервисов — это обеспечит безболезненный split.

### 5.2. Модули
| Модуль | Ответственность |
|---|---|
| `auth` | Аутентификация, JWT, роли (`buyer` / `seller` / `moderator` / `admin`) |
| `catalog` | Товары, категории, атрибуты, поиск, маркировка «Честный ЗНАК» |
| `orders` | Заказы, статусы, жизненный цикл |
| `payments` | Эскроу, расчёт комиссий, выплаты продавцам, фискализация (54-ФЗ) |
| `delivery` | Трекинг через внешний API, уровни продавца, география |
| `verification` | Верификация продавцов и товаров, рейтинг доверия |
| `reviews` | Отзывы, анти-накрутка, верификация покупки |
| `analytics` | Метрики продавца, дашборды |
| `notifications` | Push (FCM + APNs), email (SMTP), SMS (опционально) |
| `admin` | Модерация, управление верификацией, споры |

**Правило:** модули общаются **только через явные интерфейсы**. Импорт приватных пакетов одного модуля (`domain/`, `repository/`, `service/`, `external/`, `config/`) из другого модуля **запрещён** — только через `api/` подпакет (публичный контракт). Контроль — через Spring Modulith verify() в архитектурном тесте (см. [ADR-006](docs/adr/ADR-006-modular-monolith.md)).

### 5.3. Compliance-критичное (RU-специфика)

Эти три пункта **отсутствовали** в исходном ТЗ или были смягчены — фиксирую явно:

#### 5.3.1. 54-ФЗ — фискализация чеков
**Обязательно**, не опционально. Каждый платёж от физлица → чек через ОФД.
Реализация: ЮKassa.Чеки (включается на стороне ЮKassa, но `payments` модуль **обязан** передавать корректный состав чека: позиции, ставки НДС, признак способа расчёта).

#### 5.3.2. «Честный ЗНАК» — обязательная маркировка
Для целевого сегмента **обязательна** маркировка Data Matrix:
- Одежда (с 2024)
- Обувь (с 2020)
- Ювелирные изделия (с 2024)
- Постельное бельё, полотенца — отдельные категории текстиля

**Следствие для data model:**
- `Product` имеет поле `requires_marking BOOLEAN` (вычисляется по категории)
- При `requires_marking=true` создание товара без `mark_code` (Data Matrix) — **запрещено**
- При создании заказа маркировочные коды **резервируются**, при отгрузке — **передаются** в «Честный ЗНАК» через API маркировки

#### 5.3.3. 152-ФЗ — персональные данные
- Хранение данных РФ-пользователей **только** на серверах в РФ (Yandex Cloud, Selectel, VK Cloud)
- Согласие на обработку ПДн — отдельная сущность `UserConsent` с версионированием
- Право на удаление: soft delete + анонимизация в течение 30 дней

### 5.4. Data model — верхний уровень

> **Полная ERD — [/docs/erd.dbml](docs/erd.dbml).** Здесь — верхнеуровневый обзор.

Базовые сущности и их назначение:

```

auth.refresh_tokens     — refresh-токены с ротацией (JWT, CLAUDE.md 13.1)
auth.user_consents     — согласия пользователя версионируемые (152-ФЗ)

verification.sellers   — продавцы ИП/ООО (legal_form, inn, ogrn, trust_level)
verification.verification_documents — документы для верификации
verification.product_verifications — выборочная + краудсорсинг проверка товаров

catalog.categories      — категории, commission_base, requires_marking
catalog.products        — товары, цена, stock, requires_marking, gtin
catalog.product_images  — фото товаров (S3-ключи)
catalog.mark_codes      — Data Matrix коды для Честного ЗНАКа (5.3.2)

orders.addresses        — адреса доставки покупателей
orders.carts/cart_items — корзина
orders.orders           — заказы с lifecycle (см. 5.5)
orders.order_items      — позиции заказа с price_snapshot

payments.payments       — платежи с idempotency_key (CLAUDE.md 13.5)
payments.fiscal_receipts/items — фискальные чеки 54-ФЗ (ADR-008)
payments.commission_log — immutable audit расчёта комиссий (ADR-004, ADR-007)
payments.payouts        — выплаты продавцам
payments.refunds        — возвраты

delivery.deliveries     — доставки с carrier и tracking_number
delivery.tracking_events — события трекинга от курьеров (ADR-005)

reviews.reviews         — отзывы (verified_purchase=true всегда)
reviews.review_photos   — фото к отзывам
```

Каждый модуль = одна схема PostgreSQL (см. ADR-007). Это упрощает будущий split на микросервисы.

### 5.5. Жизненный цикл заказа (state machine)

```
                                    ┌──────────────────────────────────────┐
                                    ↓                                      │
created → paid → confirmed → shipped → in_transit → delivered → completed
   │       │         │                                  │
   │       │         │                                  ├→ disputed → resolved (refund_buyer | release_to_seller | partial_refund)
   │       │         │                                  └→ return_requested → returned → refunded
   │       │         │
   │       └─────────┴→ cancelled  (отмена до confirmed; для paid создаётся refund)
   └→ cancelled  (никогда не оплаченный, истечение TTL ≥ 24 часа)
```

**Важное уточнение (исправление к v1.0):** возврат `returned` возможен **только** из статусов `delivered` (через `return_requested`) или `disputed` (по решению диспута). Из `shipped` или `in_transit` товар не «возвращается», а в худшем случае «возвращается отправителю курьером» — это отдельный курьерский статус, не статус заказа.

Все переходы — **только** через `orders.api.OrderService.transitionStatus(orderId, newStatus, actor)`. Прямой UPDATE статуса в БД запрещён. Каждый переход записывается в `orders.status_transitions` (append-only audit).

### 5.6. ADR (Architecture Decision Records)
Каждое нетривиальное решение → `/docs/adr/ADR-NNN-kebab-case-title.md`.
Формат: **Контекст → Решение → Последствия → Альтернативы**.
Изменение решения = **новый** ADR со ссылкой `Supersedes: ADR-NNN`.

Обязательные ADR на Фазе 0:
- `ADR-001` — Выбор языка бэкенда (Go vs Java)

- `ADR-003` — Выбор очереди сообщений
- `ADR-004` — Правила суммирования бонусов комиссии (см. OPEN_QUESTION в разделе 3)
- `ADR-005` — Выбор агрегатора трекинга (AfterShip vs TrackingMore)
- `ADR-006` — Модульный монолит как стартовая архитектура
- `ADR-007` — Стратегия миграций БД и совместимости
- `ADR-008` — Стратегия фискализации (54-ФЗ)

  ```go
  if attempts > 5 { ... }
  // ✅ Хорошо
  const MaxLoginAttempts = 5
  if attempts > MaxLoginAttempts { ... }
  ```
- **Без скрытых зависимостей.** Dependency injection везде. Конструктор/функция явно принимает зависимости.
- **Без паники в продакшен-коде** (Go): все ошибки обрабатываются явно через `error`.
- **Без `System.exit()`** в продакшен-коде (Java).
- **SQL только через миграции** — никогда вручную, никогда `ALTER` в проде через `psql`.
- **Секреты только через переменные окружения** — никогда в коде, никогда в `git`.
- **Структурированное логирование** (JSON, поля `level`, `ts`, `module`, `trace_id`, `user_id`).

### 8.2. Go-специфика (если выбран Go)
- `gofmt` + `goimports` обязательны (pre-commit hook)
- `golangci-lint` со стандартным конфигом проекта (`.golangci.yml`)
- Контекст `context.Context` — **первый аргумент** во всех методах, выполняющих I/O
- Ошибки оборачиваются через `fmt.Errorf("operation: %w", err)` — никогда не теряются
- Интерфейсы определяются на стороне **потребителя**, не провайдера

### 8.3. Java-специфика (если выбран Java)
- Spotless + Google Java Format
- Records для DTO, не Lombok-классы
- `@Transactional` только в сервисном слое, никогда в контроллерах
- Никаких `Optional.get()` без предварительной проверки `isPresent()`
- Конструкторная инъекция (final поля), не `@Autowired` на полях

### 8.4. Flutter / Dart
- `flutter analyze` без warning'ов — обязательно
- Архитектура: **Riverpod** для state management (фиксируется отдельным ADR на Фазе 4)
- Модели — через `freezed` + `json_serializable`
- Никакого бизнес-логики в виджетах

### 8.5. Git и коммиты (Conventional Commits)
- `feat:` — новая функциональность
- `fix:` — баг
- `refactor:` — без изменения поведения
- `test:` — тесты
- `docs:` — документация
- `chore:` — инфраструктура, зависимости
- `ci:` — CI/CD
- `perf:` — оптимизация производительности

Связь с задачей **обязательна**:
```
feat(catalog): add product marking validation

Closes TASK-042
```

### 8.6. Ветвление (Trunk-based)
- `main` — всегда деплоябельна
- `feature/TASK-NNN-short-desc` — фича-ветки, **не дольше 2 дней**
- `hotfix/brief-desc` — хотфиксы
- Squash merge в `main`
- Long-living ветки запрещены

### 8.7. Pull Request
- Один PR = одна задача из `/docs/tasks/`
- Шаблон описания:
  ```
  ## Что сделано
  ## Как проверить
  ## Связанная задача
  ## Чеклист
  - [ ] Тесты добавлены
  - [ ] OpenAPI обновлён (если API менялся)
  - [ ] Миграция БД создана (если схема менялась)
  - [ ] CHANGELOG обновлён (если breaking change)
  ```
- Минимум 1 ревью (когда в команде >1 человека); для соло-разработки — самопроверка по чеклисту

### 8.8. Definition of Done
Задача считается **выполненной**, когда:
- ☐ Код написан и проходит линтер
- ☐ Unit-тесты написаны и проходят
- ☐ Integration-тесты (если применимо) написаны и проходят
- ☐ Покрытие бизнес-логики ≥ 80%
- ☐ API задокументирован в OpenAPI (для backend-задач)
- ☐ Миграция БД создана (если схема менялась)
- ☐ Описание PR заполнено
- ☐ Нет `TODO` / `FIXME` без ссылки на тикет

## 12. Стоп-условия

Claude **обязан остановиться** и явно спросить пользователя при любом из этих условий:

1. **Изменение схемы БД** для уже реализованного и работающего модуля

## 13. Безопасность

### 13.1. Аутентификация и авторизация
- **JWT** с refresh-tokens
- Access token TTL: **≤ 15 минут**
- Refresh token TTL: 30 дней, ротация при использовании
- RBAC: `buyer` / `seller` / `moderator` / `admin` — проверка на **каждом** endpoint
- Пароли: **argon2id** (предпочтительно) или **bcrypt** с cost ≥ 12
- Минимальная длина пароля: 12 символов
- Проверка на утечки через `haveibeenpwned` API при регистрации и смене пароля

### 13.2. Транспорт и заголовки
- HTTPS everywhere, **HSTS** включён
- CORS — explicit allow-list, никаких `*` в продакшене
- Security headers: `X-Frame-Options`, `X-Content-Type-Options`, `Content-Security-Policy`

### 13.3. Валидация входных данных
- На **всех** входящих данных: размер, тип, формат, диапазон
- SQL — **только** параметризованные запросы (никаких `fmt.Sprintf` в SQL)
- Загрузка файлов:
  - Проверка MIME type (не доверять `Content-Type` из запроса — определять по содержимому)
  - Лимит размера: **10 МБ** на изображение, **50 МБ** на видео
  - Хранение **вне** webroot (S3 / MinIO)
  - Сканирование на вредоносный контент (опционально, через ClamAV)

### 13.4. Rate limiting

Учёт ведётся по двум измерениям параллельно (срабатывает любой из лимитов):

- **Auth endpoints (`/auth/login`):**
  - **5 попыток / минуту на IP** — защита от bot-атак
  - **5 попыток / минуту на login** — защита целевого аккаунта при атаке с нескольких IP/NAT-сетей
- **Регистрация (`/auth/register`):** 3 / час на IP
- **API в целом:** 100 / минуту на аутентифицированного пользователя (sliding window)
- **Анонимные публичные эндпоинты** (GET /products, /categories): 60 / минуту на IP

После исчерпания лимита возвращается `429 Too Many Requests` с заголовком `Retry-After` (секунды). Реализация — через Resilience4j Rate Limiter, ключи в Redis для распределённости.

### 13.5. Финансовая безопасность
- **Эскроу** обязателен — деньги не уходят продавцу до подтверждения получения
- **Идемпотентность** всех платёжных операций (заголовок `Idempotency-Key`)
- **Аудит-лог** всех финансовых транзакций — **immutable**, write-only таблица `commission_log` / `payment_log`
- **Двойная проверка** суммы комиссии (клиент рассчитывает для отображения, сервер — для списания, расхождение → отказ)

### 13.6. Compliance (RU)
- **152-ФЗ:** согласие пользователя (версионируется), право на удаление, хранение данных РФ-граждан в РФ
- **54-ФЗ:** фискализация чеков через ОФД (обязательно)
- **Маркировка «Честный ЗНАК»:** для одежды, обуви, ювелирных изделий, текстиля

### 13.7. Безопасность мобильных приложений
- **Certificate pinning** для API запросов
- **Biometric auth** опционально (Face ID / Touch ID / Android Biometric)
- Чувствительные данные — в **Keychain** (iOS) / **EncryptedSharedPreferences** (Android), не в plain SharedPreferences/UserDefaults
- **ProGuard / R8** для Android release builds
- **Disable backup** для приложения продавца (предотвращает утечку credentials через iCloud/Google backup)

## 16. Формат технического задания (TASK-NNN.md)

Каждая задача описывается в едином YAML-формате. Файлы хранятся в `/docs/tasks/TASK-NNN.md`.

### Шаблон TASK-NNN.md

```yaml
id: TASK-042
title: "Добавить валидацию маркировки товара при создании"
phase: 1
module: catalog
priority: high
depends_on: [TASK-001, TASK-008]
estimated_hours: 8

context: |
  Согласно требованиям 54-ФЗ и системе маркировки «Честный ЗНАК», товары из категорий
  с обязательной маркировкой (одежда, ювелирные изделия, обувь) не могут быть созданы
  без Data Matrix кода маркировки. Это критично для compliance.

acceptance_criteria:
  - given: "продавец создаёт товар в категории 'Одежда'"
    when: "не указан mark_code"
    then: "API возвращает 400 Bad Request с сообщением об ошибке"
  
  - given: "товар имеет requires_marking=true (по категории)"
    when: "mark_code валидный и уникальный"
    then: "товар успешно создаётся и код резервируется в системе маркировки"
  
  - given: "товар из категории без маркировки (например, 'Товары для дома')"
    when: "mark_code не указан"
    then: "товар создаётся без ошибок"

technical_notes: |
  - Валидация на стороне API (catalog.ProductService.CreateProduct)
  - Проверка requires_marking в таблице Category
  - При requires_marking=true, mark_code должен быть отправлен в API маркировки
    для резервирования (см. раздел 5.3.2 CLAUDE.md)
  - В случае ошибки резервирования откатить транзакцию создания товара
  - Требуется новая миграция БД для поля mark_code (если ещё не добавлено)

api_changes:
  - method: POST
    path: /api/v1/products
    description: "Создание товара с валидацией маркировки"
  
  - method: GET
    path: /api/v1/categories/{id}
    description: "Поле requires_marking должно быть в ответе"

db_changes:
  - type: migration
    description: "Добавить поле mark_code (VARCHAR, UNIQUE) в таблицу products"
  
  - type: migration
    description: "Добавить поле requires_marking (BOOLEAN DEFAULT false) в таблицу categories"

test_requirements:
  unit:
    - "ProductService.CreateProduct возвращает ошибку при requires_marking=true и mark_code=null"
    - "ProductService.CreateProduct успешен при requires_marking=true и валидном mark_code"
    - "ProductService.CreateProduct успешен при requires_marking=false и mark_code=null"
    - "Расчёт requires_marking по category.id (table-driven test с разными категориями)"
  
  integration:
```

### Правила заполнения

| Поле | Обязательно | Примечание |
|---|---|---|
| `id` | Да | Формат: `TASK-NNN` (трёхзначное число) |
| `title` | Да | Краткое название, начиная с глагола |
| `phase` | Да | Одна из: 0, 1, 2, 3, 4, 5 |
| `module` | Да | Из списка модулей в разделе 5.2 |
| `priority` | Да | critical / high / medium / low |
| `depends_on` | Нет | Список TASK-NNN, которые должны быть завершены перед этой |
| `estimated_hours` | Да | Реалистичная оценка с запасом |
| `context` | Да | Бизнес-обоснование, почему это важно |
| `acceptance_criteria` | Да | Минимум 3 сценария (given-when-then) |
| `technical_notes` | Да | Детали реализации, ограничения, ссылки на ADR |
| `api_changes` | Если API | Все endpoints, которые меняются или создаются |
| `db_changes` | Если БД | Все миграции и их назначение |
| `test_requirements` | Да | Разделённо: unit и integration тесты |
| `definition_of_done` | Да | Чеклист из раздела 8.8, адаптированный под задачу |

### Когда создавать файл TASK-NNN.md

**Правило (зафиксировано 2026-05-07):** TASK-файлы создаются **по требованию, непосредственно перед началом работы** — не оптом заранее.

Причины:
- Оптовое создание 17 файлов на фазу = 17 документов, которые устаревают за 1-2 недели по мере уточнения скоупа.
- Точные `acceptance_criteria` и `technical_notes` пишутся **после** прочтения свежего состояния кода и связанных ADR, а не из абстрактных представлений.
- Оценка `estimated_hours` достоверна максимум на 1-2 ближайшие задачи; дальше — фантазия.

**Что должно существовать заранее:**
- Запись в `docs/tasks/INDEX.md` (id, title, статус 🔵 TODO) — это roadmap, не ТЗ.
- Зависимости (`depends_on`) — чтобы видеть критический путь фазы.

**Что создаётся в момент начала работы:**
- Полный файл `TASK-NNN.md` со всеми обязательными полями.
- Ветка `feature/TASK-NNN-short-desc`.

**Что обновляется при завершении:**
- Статус в `INDEX.md`: 🔵 TODO → 🟡 IN_PROGRESS → 🟢 DONE.
- В самом TASK-файле — добавляется `status: completed` и `completed_at`.

**Исключение:** если задача — продолжение свежей итерации (например, `make adr-new` мы создали в одной сессии без отдельного TASK), допустимо документировать постфактум одним коротким TASK-NNN-retrospective.md. Не злоупотреблять.
