# ADR-005 — Стратегия трекинга доставки

- **Статус:** Accepted
- **Дата:** 2026-05-07
- **Автор:** Tech Lead

## Контекст

CLAUDE.md раздел 4 предлагает использовать AfterShip или TrackingMore как агрегатор трекинга. Оба сервиса — иностранные (AfterShip — Гонконг/США, TrackingMore — Китай).

Возникает конфликт с CLAUDE.md раздел 5.3.3 / 13.6 (152-ФЗ): передача трек-номера в связке с покупательским заказом (даже без явного ФИО) считается передачей ПДн третьим лицам и **трансграничной передачей**, что требует:
- Раздельного согласия пользователя на трансграничную передачу.
- Информирования о юрисдикции получателя.
- Или хранения данных в РФ.

Дополнительно — санкционные риски: в 2022–2025 было несколько прецедентов отключения российских мерчантов от иностранных сервисов трекинга.

## Решение

**На старте (Фазы 1–3): прямые интеграции с API российских курьеров через единый абстрактный слой `delivery.tracking.CarrierTrackingClient`.**

Минимальный набор поддерживаемых перевозчиков на старте:
1. **СДЭК** — публичный API V2 (REST).
2. **Почта России** — Tracking API (через личный кабинет отправителя).
3. **Boxberry** — REST API.
4. **Яндекс Доставка** — API (по договору).

При необходимости — добавление через адаптер без изменения сервисного слоя.

## Обоснование

| Критерий | Прямые API курьеров | AfterShip/TrackingMore |
|---|---|---|
| Соответствие 152-ФЗ | ✅ Данные в РФ | ❌ Трансграничная передача |
| Устойчивость к санкциям | ✅ | ❌ Был кейс отключений |
| Покрытие РФ-курьеров | ✅ Прямой доступ | ⚠️ Через интеграции, иногда с задержкой обновлений |
| Глубина данных | ✅ Полные события из системы курьера | ⚠️ Стандартизованная схема, иногда неполная |
| Сложность разработки | ⚠️ N интеграций (4+) | ✅ Одна интеграция |
| Стоимость | ✅ Бесплатно (в большинстве случаев) | ⚠️ От $50/мес и выше |
| Webhooks vs polling | ⚠️ Зависит от курьера (СДЭК — webhooks, Почта — только polling) | ✅ Унифицированно |

Сложность разработки выше, но это разовая инвестиция Фазы 3, и она снимает оба ключевых риска (compliance + санкции).

Контракт интерфейса:
```java
public interface CarrierTrackingClient {
    Carrier carrier();
    TrackingSnapshot fetchSnapshot(String trackingNumber);   // synchronous polling
    boolean supportsWebhooks();
    Optional<TrackingSnapshot> parseWebhook(WebhookPayload payload);
}
```

Polling-стратегия:
- Активные заказы (`shipped`, `in_transit`) — раз в 6 часов.
- Доставленные / отменённые — больше не опрашиваются.
- Webhook'и (где доступны) — приоритетнее polling'а; polling выступает fallback'ом.

Все опросы и обработка вебхуков идут через очередь `delivery.tracking-poll.v1` в RabbitMQ (см. ADR-003).

## Compliance

- Согласие на передачу ПДн **российскому** курьеру включается в общие согласия пользователя при оформлении заказа.
- Отсутствует трансграничная передача → не требуется отдельное согласие.
- Все вызовы курьерских API логируются (без чувствительных данных) для audit-trail.

## Последствия

### Положительные
- Полное соответствие 152-ФЗ — данные не покидают РФ.
- Устойчивость к санкционным рискам (отключения иностранных сервисов).
- Глубокие нативные события от каждого курьера, без приведения к lowest-common-denominator-схеме агрегатора.
- Нулевая стоимость API-доступа (большинство публичных API).

### Отрицательные / риски
- 4 интеграции вместо одной — больше первоначальной работы (компенсируется единым интерфейсом `CarrierTrackingClient`).
- Polling для Почты России (нет webhook'ов) — нагрузка на их API и задержки в обновлении статусов.
- При смене API курьера ломается только одна реализация — но требуется поддержка адаптеров на длинной дистанции.

### Что меняется в проекте
- Добавляется пакет `ru.truemarket.delivery.tracking` с интерфейсом `CarrierTrackingClient`.
- Добавляется enum `Carrier` (CDEK / RUSSIAN_POST / BOXBERRY / YANDEX / PICKUP / OTHER) — отражён в ERD.
- В очередь RabbitMQ добавляется `delivery.tracking-poll.v1` (опрос неактивных деливери).
- Контроллер `CdekWebhookController` (и аналогичные для остальных, где доступны вебхуки).

## Триггеры пересмотра

Этот ADR может быть пересмотрен в случае:
- Появления международных продавцов (нужны зарубежные курьеры).
- Перехода на международный рынок.
- Появления отечественного сертифицированного агрегатора с покрытием 4+ перевозчиков на условиях, лучше прямых интеграций.

## Альтернативы

- **AfterShip с явным согласием на трансграничную передачу** — отклонён по совокупности (санкционный риск + операционный риск отключения).
- **TrackingMore** — то же самое.
- **Делать только трекинг по СДЭК на старте, остальные — позже** — рассмотрен. Принят как фаллбек: если объём работ по 4 курьерам не помещается в Фазу 3, оставляем СДЭК + Почту России как обязательные, остальные — Фаза 5.
