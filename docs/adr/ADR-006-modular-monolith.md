# ADR-006 — Модульный монолит как стартовая архитектура

- **Статус:** Accepted
- **Дата:** 2026-05-07
- **Автор:** Tech Lead

## Контекст

CLAUDE.md раздел 5.1 фиксирует выбор: **модульный монолит на старте → миграция к микросервисам по мере роста**. Этот ADR уточняет механики и контроль соблюдения границ модулей, который **обязателен** для будущего безболезненного раздела на сервисы.

## Решение

1. На старте — единый Spring Boot приложение, разбитое на **доменные модули** в иерархии пакетов `ru.truemarket.<module>`.
2. Контроль границ модулей — через **Spring Modulith** (`org.springframework.modulith:spring-modulith-starter-core`).
3. Внутримодульная архитектура — слоёная (api / service / domain / repository / external).
4. Межмодульные взаимодействия — **только через явные публичные интерфейсы** в `api/` подпакете и через **доменные события** (Spring Application Events).
5. Каждый модуль имеет `package-info.java` с аннотацией `@ApplicationModule`.
6. Архитектурные тесты на границы модулей — обязательны, запускаются в CI.

## Структура модуля

```
ru.truemarket.<module>/
├── package-info.java          // @ApplicationModule(displayName = "<Module>", allowedDependencies = ...)
├── api/                       // ПУБЛИЧНЫЙ слой: DTO + сервис-интерфейсы
│   ├── <Module>Service.java
│   └── dto/...
├── domain/                    // ПРИВАТНЫЙ слой: сущности, value objects
├── service/                   // ПРИВАТНЫЙ слой: реализация сервисов
├── repository/                // ПРИВАТНЫЙ слой: JPA-репозитории
├── external/                  // ПРИВАТНЫЙ слой: клиенты внешних API
└── config/                    // ПРИВАТНЫЙ слой: @Configuration
```

Spring Modulith автоматически обеспечивает: внутренние пакеты модуля **не доступны** другим модулям; разрешён импорт только из `api/`.

## Доменные события

Бизнес-логика, требующая реакции из другого модуля, публикует событие через `ApplicationEventPublisher`:

```java
// в orders.service.OrderService
events.publishEvent(new OrderPaidEvent(order.id(), order.total(), order.buyerId()));
```

Подписчик в другом модуле:
```java
// в notifications.service.OrderEventListener
@ApplicationModuleListener   // обеспечивает асинхронную транзакционную обработку
public void on(OrderPaidEvent event) { ... }
```

`@ApplicationModuleListener` гарантирует:
- Асинхронную обработку (новый поток).
- Изоляцию транзакций — отправитель не «висит» в ожидании подписчиков.
- Возможность дальнейшего вынесения подписчика в отдельный сервис без изменения кода — событие просто будет доставлено через RabbitMQ (см. ADR-003).

## Контроль границ модулей в CI

Архитектурные тесты обязательны:
```java
@Test
void verifyModularStructure() {
    var modules = ApplicationModules.of(TrueMarketApplication.class);
    modules.verify();   // упадёт, если модули нарушают правила
}
```

При появлении нелегального импорта `internal/x` из `internal/y` — CI блокирует merge.

## Карта модулей и допустимых зависимостей

| Модуль | Что делает | Зависит от (через `api/`) |
|---|---|---|
| `auth` | Аутентификация, JWT, RBAC | — |
| `catalog` | Товары, категории, поиск | `auth` |
| `orders` | Заказы, статусы, lifecycle | `auth`, `catalog` |
| `payments` | Эскроу, комиссии, фискализация | `orders` |
| `delivery` | Трекинг, уровни, география | `orders` |
| `verification` | Верификация продавцов, товаров | `auth`, `catalog` |
| `reviews` | Отзывы, анти-накрутка | `auth`, `orders`, `catalog` |
| `analytics` | Метрики продавца | `orders`, `payments` (read-only) |
| `notifications` | Push, email, sms | (слушает события всех) |
| `admin` | Модерация, споры | `auth`, `verification`, `orders`, `reviews` |
| `common` | Утилиты, общие типы | — (никаких бизнес-зависимостей) |

**Циклические зависимости запрещены.** При попытке создать цикл — Modulith упадёт в архитектурном тесте.

## Стратегия будущей миграции на микросервисы

Сценарий выноса модуля X в отдельный сервис:
1. `api/` модуля X уже представляет публичный контракт.
2. Все межмодульные вызовы заменяются на REST/gRPC-клиент с тем же интерфейсом.
3. Подписки на события через `@ApplicationModuleListener` заменяются на consumer'ы RabbitMQ-очереди.
4. Извлекаются миграции БД модуля X в отдельную схему PostgreSQL → отдельная БД.

Никаких `JOIN` между модулями на уровне SQL **запрещены изначально** — это самое частое препятствие для выноса.

## Триггеры выноса в отдельный сервис

- Профиль нагрузки одного модуля принципиально отличается от остальных (например, `delivery` — много I/O wait, мало CPU).
- Команда из >1 человек хочет деплоить модуль независимо.
- Модулю нужны разные SLA / разные регламенты безопасности.
- Размер репозитория / время CI начали мешать продуктивности.

## Последствия

### Положительные
- Скорость разработки MVP.
- Простой деплой (один контейнер).
- Атомарные транзакции в пределах одного модуля без распределённых транзакций.
- Безболезненный split при необходимости.

### Отрицательные / риски
- Соблазн «срезать угол» через прямой доступ к таблицам другого модуля. Митигация: архитектурные тесты в CI + правило «никаких межмодульных JOIN».
- Один deploy-юнит — деплой одного изменения требует рестарта всего.

## Альтернативы

- **Сразу микросервисы** — отклонён: преждевременная оптимизация, замедлит MVP в 2–3×, потребует распределённых транзакций для эскроу-сценария до того, как у нас есть пользователи.
- **Hexagonal architecture без модулей** — отклонён: не даёт явных границ для будущего split'а.
