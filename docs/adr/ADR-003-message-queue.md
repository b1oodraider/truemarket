# ADR-003 — Выбор очереди сообщений

- **Статус:** Accepted
- **Дата:** 2026-05-07
- **Автор:** Tech Lead

## Контекст

CLAUDE.md раздел 4 предлагает выбор: **RabbitMQ** или **Kafka**.

Сценарии использования очередей в TrueMarket:
- Уведомления (push, email) — fire-and-forget, низкая латентность.
- Асинхронная обработка событий заказа (статусы, выплаты).
- Резервирование/передача маркировочных кодов в «Честный ЗНАК» с ретраями.
- Webhook'и от ЮKassa — отдельная очередь с DLQ.
- Audit-event log — append-only поток (потенциально подходит для Kafka).

## Решение

**RabbitMQ 3.13+ с плагином `rabbitmq_management` (для admin UI) и `rabbitmq_prometheus` (для метрик).**

**Отложенные сообщения** реализуются нативным механизмом **TTL + Dead-Letter Exchange (DLX)** —
без community-плагина `rabbitmq_delayed_message_exchange`. Это намеренный выбор: плагин не входит в
стандартный образ Docker Hub `rabbitmq:management-alpine`, а тащить custom Dockerfile ради этой
функциональности в Phase 0 — преждевременная сложность. Если в Phase 3+ появится массовый сценарий
с большой кардинальностью delayed-сообщений (один tick на каждый out-of-stock уведомитель и т. п.),
заведём отдельный ADR с обоснованием перехода на плагин.

Stream-нагрузки (audit-log, аналитика реального времени) на старте идут через ту же RabbitMQ; при появлении настоящего event-streaming use-case (например, CDC для search-индекса или ML-feature-store) принимается отдельный ADR на Kafka / Redpanda.

## Обоснование

| Критерий | RabbitMQ | Kafka |
|---|---|---|
| Семантика queue (work queues) | ✅ Нативная, простая | ⚠️ Через consumer-groups, не идеально |
| Семантика pub/sub | ✅ Топики/exchanges | ✅ Топики |
| Семантика event-streaming (replay, длинная история) | ❌ Не предназначена | ✅ Нативная |
| Объём операций RPS | До 50 000+ | Миллионы |
| Latency p99 | ✅ Единицы мс | ⚠️ 10–100 мс (зависит от настроек) |
| Сложность операционной поддержки | ✅ Один бинарник | ⚠️ Kafka + ZooKeeper / KRaft, больше тюнинга |
| Стоимость в проде | ✅ 1 нода 2 ГБ RAM | ⚠️ Минимум 3 ноды для HA |
| Готовность Spring (Spring AMQP / Spring Kafka) | ✅ Зрелая | ✅ Зрелая |
| Delayed messages | ✅ Через плагин или TTL+DLX | ⚠️ Не из коробки |
| DLQ (dead letter queue) | ✅ Нативный | ⚠️ Через топики |

Для **MVP** маркетплейса нужны: work queues (notifications, marking codes), pub/sub (события заказов между модулями), DLQ, retry с задержкой. Всё это RabbitMQ покрывает с минимальной сложностью.

Kafka станет нужна при появлении event-sourcing или streaming-аналитики — не на Фазе 0.

## Последствия

### Положительные
- Один контейнер в Docker Compose, 5 минут на разворачивание.
- `spring-boot-starter-amqp` — готовая интеграция.
- Простое отслеживание очередей через RabbitMQ Management UI.
- DLQ из коробки, retry с экспоненциальной задержкой через DLX.

### Отрицательные / риски
- Не подходит для replay событий за длительный период — при появлении такого требования вводится отдельный Kafka-кластер.
- Менее эффективна для очень больших batch'ей.

### Архитектурные правила
1. Имена очередей: `<module>.<event-or-command>.<version>`, например `orders.status-changed.v1`, `notifications.send-push.v1`.
2. Все очереди — **durable**, все сообщения — **persistent**.
3. У каждой очереди есть **DLQ** и политика максимального числа ретраев (по умолчанию 5).
4. Сериализация — **JSON** (через Jackson) с явной схемой DTO. На длинной дистанции — миграция на Avro/Protobuf для строгой эволюции схем.
5. Idempotency-ключ обязателен для всех команд (не событий).

## Альтернативы

- **Kafka / Redpanda** — отложен. Триггеры для возврата:
  - Появление CDC для индексации каталога в search.
  - Audit-log объёмом > 10 ГБ/день с требованием replay.
  - Event-sourcing для одного из доменов.
- **Redis Streams** — рассмотрен как нулевая опция (Redis уже в стеке для кэша/сессий). Отклонён: смешивание ролей кэша и очереди создаёт операционные риски (eviction-политики могут уничтожить сообщения).
- **NATS / NATS JetStream** — простой, но менее распространённый. Отклонён: ниже знакомство в Java/Spring сообществе.
