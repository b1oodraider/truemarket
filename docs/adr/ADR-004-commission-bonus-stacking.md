# ADR-004 — Правила суммирования бонусов комиссии

- **Статус:** Accepted
- **Дата:** 2026-05-07
- **Автор:** Tech Lead (по делегированию решения от Product Owner)

## Контекст

CLAUDE.md раздел 3 описывает две перекрывающиеся структуры скидок к комиссии:

**Бонусы (суммируются, max −6%):**
- Российское производство: −2%
- Сертификация (ГОСТ/ТР ТС): −1%
- **Быстрая доставка (уровень Trusted+): −1.5%**
- Верифицированный продавец: −1.5%

**Уровни продавца по географии:**
- `Verified Local` — базовая
- **`Trusted Seller` (ЕЧР ≤ 7 дней) — −1.5% + badge**
- `National` — −3% + приоритет в выдаче
- `All-Russia` — −4.5% + featured

Открытый вопрос (OQ-001 / TASK-004 в плане): бонус «Быстрая доставка (Trusted+)» из первой группы и автоматическая скидка уровня `Trusted Seller` из второй группы — **это одно и то же** (не суммируется) **или два независимых вычета** (суммируется)?

## Решение

**Бонус «Быстрая доставка (Trusted+)» и автоматический вычет уровня `Trusted Seller` — это один и тот же бонус, не складываются.**

Реализация в коде: **универсальная функция расчёта комиссии получает множество бонусов и применяет их с дедупликацией**. Бонус «Быстрая доставка» технически вычисляется через достижение уровня `Trusted Seller` или выше.

## Обоснование

1. **Интуиция бизнеса.** Уровень `Trusted Seller` присваивается **только** продавцам с ЕЧР ≤ 7 дней — то есть «быстрая доставка» — это и есть критерий уровня. Дублировать вычет за тот же критерий было бы для продавца неожиданным двойным выигрышем.
2. **Прогнозируемость экономики.** Если бы скидки складывались, минимальная комиссия по одежде получилась бы 12% − (2 + 1 + 1.5 + 1.5 + 4.5) = **1.5%**, что ниже декларированного минимума 6%. Текущий жёсткий потолок −6% был бы постоянно достижим, что разрушает стимулы для вхождения в верхние уровни.
3. **Совместимость с верхними уровнями.** Уровни `National` (−3%) и `All-Russia` (−4.5%) **тоже включают** быструю доставку как поддоминантное условие. Считаем, что эти три уровня — взаимно исключающие, продавец получает **только один** уровень-вычет.
4. **Гипотеза по умолчанию из CLAUDE.md.** В разделе 3 CLAUDE.md уже зафиксирована эта гипотеза («один и тот же бонус, не складывается»). Тестовые сценарии расчёта комиссии должны это проверять.

## Формальные правила расчёта (для реализации)

```
Дано:
  base_rate                — базовая ставка категории (например, 0.12 для одежды)
  delivery_tier            — один из: VERIFIED_LOCAL, TRUSTED, NATIONAL, ALL_RUSSIA
  is_russian_made          — boolean (российское производство)
  has_certification        — boolean (ГОСТ/ТР ТС)
  is_verified_seller       — boolean (верифицированный продавец)

Шаг 1. Скидка за уровень доставки (взаимно исключающие):
  level_discount =
    case delivery_tier of
      VERIFIED_LOCAL → 0.000
      TRUSTED        → 0.015   // та же скидка, что и «быстрая доставка»
      NATIONAL       → 0.030
      ALL_RUSSIA     → 0.045

Шаг 2. Дополнительные бонусы (суммируются между собой):
  extra_bonuses = 0
  if is_russian_made:    extra_bonuses += 0.020
  if has_certification:  extra_bonuses += 0.010
  if is_verified_seller: extra_bonuses += 0.015

Шаг 3. Суммарная скидка с потолком −6%:
  total_discount = min(level_discount + extra_bonuses, 0.060)

Шаг 4. Финальная ставка:
  final_rate = max(base_rate - total_discount, 0)

Шаг 5. Минимальная фактическая комиссия (защита от ошибок конфигурации):
  final_rate = max(final_rate, 0.040)   // абсолютный минимум 4%
```

### Примеры

| Сценарий | base | level | extras | total_discount (после капа) | final_rate |
|---|---|---|---|---|---|
| Одежда, Verified Local, ничего | 12% | 0% | 0% | 0% | 12% |
| Одежда, Trusted, без бонусов | 12% | 1.5% | 0% | 1.5% | 10.5% |
| Одежда, Trusted, РФ + сертификат + верификация | 12% | 1.5% | 4.5% | 6% (cap) | 6% |
| Одежда, All-Russia, РФ + сертификат + верификация | 12% | 4.5% | 4.5% | 6% (cap) | 6% |
| Ювелирка, All-Russia, РФ + сертификат + верификация | 10% | 4.5% | 4.5% | 6% (cap) | 4% |
| Дом, Trusted, нет других бонусов | 10% | 1.5% | 0% | 1.5% | 8.5% |

Минимальная итоговая комиссия:
- Одежда / текстиль (12%) — **6%**
- Ювелирка / товары для дома (10%) — **4%**

Эти значения совпадают с целевыми из CLAUDE.md раздел 3.

## Реализация в коде

```java
public record CommissionInput(
    BigDecimal baseRate,
    DeliveryTier tier,
    boolean isRussianMade,
    boolean hasCertification,
    boolean isVerifiedSeller
) {}

public record CommissionResult(
    BigDecimal baseRate,
    BigDecimal levelDiscount,
    BigDecimal extraBonuses,
    BigDecimal totalDiscount,
    BigDecimal finalRate,
    Map<String, BigDecimal> bonusBreakdown   // для аудита
) {}

public CommissionResult calculate(CommissionInput input) { ... }
```

### Аудит
Каждый расчёт комиссии при создании заказа сохраняется в `commission_log` (immutable) с полным `bonusBreakdown` в поле `bonuses_jsonb`. Это требование CLAUDE.md раздел 13.5.

## Тесты (обязательно)
- Table-driven test со всеми сочетаниями `tier × {is_russian_made, has_certification, is_verified_seller}` × `category` (одежда, ювелирка, дом) — **3 × 8 × 3 = 72 кейса**.
- Проверка границ: cap на 6%, абсолютный минимум 4%.
- Защита от мутации `commission_log` (попытка update должна падать на уровне БД).
- Проверка идемпотентности: повторный расчёт для того же заказа возвращает байт-в-байт идентичный результат.

## Последствия

### Положительные
- Прозрачные правила, понятные продавцу.
- Сохраняется бизнес-стимул для перехода на верхние уровни доставки.
- Простая модель — один источник истины для скидки за географию.

### Отрицательные / риски
- Если в будущем появятся ещё типы бонусов (например, «новый продавец», «sale-период»), потребуется расширение модели. Решается через добавление новых ключей в `bonusBreakdown`.

## Альтернативы

- **Складывать оба бонуса (вариант B)** — отклонён: создаёт двойной счёт за один и тот же критерий.
- **Использовать только уровни без extra-бонусов** — отклонён: убирает стимулы для российского производства и сертификации.
