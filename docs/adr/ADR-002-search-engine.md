# ADR-002 — Выбор поискового движка

- **Статус:** Accepted
- **Дата:** 2026-05-07
- **Автор:** Tech Lead

## Контекст

CLAUDE.md раздел 4 предлагает выбор: **Elasticsearch** или **Meilisearch**.
Контекст:
- На Фазе 1 (MVP) объём каталога ≤ 100 000 SKU, до ~50 запросов/сек.
- Базовые требования: typo-tolerance, фасеты по атрибутам (категория, цена, размер, материал), сортировка по релевантности и цене.
- На дистанции (Фаза 5+) возможны: семантический поиск, рекомендации, нечёткий поиск с учётом синонимов и морфологии русского языка.
- В команде на старте — соло-разработчик / небольшая команда, операционная нагрузка должна быть минимальной.

## Решение

**Meilisearch v1.10+ на старте (Фазы 1–3). Стратегия миграции на Elasticsearch / OpenSearch при превышении лимитов.**

## Обоснование

| Критерий | Meilisearch | Elasticsearch |
|---|---|---|
| Время разворачивания (от нуля до индекса) | ✅ Минуты | ⚠️ Часы (heap, indexing strategy, mappings) |
| Память (1 нода, 100k документов) | ✅ ~500 МБ | ⚠️ 2–4 ГБ |
| Typo-tolerance из коробки | ✅ Включена по умолчанию | ⚠️ Требует настройки fuzziness, edit distance |
| Релевантность для коротких запросов | ✅ Tuned-for-search-as-you-type | ⚠️ Нужен прокачанный analyzer |
| Морфология русского языка | ⚠️ Базовая (lemma-rules, лимит языков) | ✅ Polnyj русский анализатор (`russian` analyzer + Hunspell) |
| Фасеты, агрегации | ✅ Достаточно для каталога | ✅ Промышленный уровень |
| Полнотекст по большим документам (>10 КБ) | ⚠️ Хуже | ✅ Промышленный уровень |
| Документация и SDK для Java/Spring | ✅ Официальный Java SDK + сообщество | ✅ Spring Data Elasticsearch |
| Лицензия | ✅ MIT (Core) | ⚠️ SSPL/Elastic License (с 2021) — ограничения для облака |
| Операционная нагрузка | ✅ Один бинарник, без кластера | ⚠️ Требует понимания shards/replicas/heap |
| Стоимость хостинга | ✅ Низкая | ⚠️ 3–5× дороже на сравнимом профиле |

Meilisearch выигрывает по совокупности на Фазе 1–3. Главный риск — морфология русского — закрывается стеммингом на стороне приложения и пользовательскими словарями синонимов (Meilisearch это поддерживает).

## Последствия

### Положительные
- Минимум 1–2 недели экономии на Фазе 1.
- Низкие расходы на хостинг (один контейнер 1 ГБ RAM на старте).
- Простое локальное окружение через Docker Compose.

### Отрицательные / риски
- При росте каталога > 1 млн SKU и сложных аналитических запросах потребуется миграция.
- Нужен **абстрактный слой `catalog.search.SearchEngine`** в коде, чтобы миграция не превратилась в переписывание catalog-модуля.
- Отсутствие промышленных аналитик-функций (типа percentile-агрегаций) — для дашбордов используется отдельный путь через Postgres + материализованные представления.

### Архитектурное требование
В пакете `ru.truemarket.catalog.search` создаётся интерфейс:
```java
public interface ProductSearchEngine {
    SearchResult<ProductDoc> search(SearchQuery query);
    void indexProduct(ProductDoc doc);
    void removeProduct(UUID productId);
    void reindexAll();
}
```
Реализация `MeilisearchProductSearchEngine` — единственная на старте. Все вызовы из сервисного слоя — только через интерфейс.

## Триггеры миграции на Elasticsearch / OpenSearch

Любое из:
- Каталог > 1 млн активных SKU.
- p95 latency поиска > 300 мс на типовом запросе.
- Бизнес-требование на сложные агрегации (например, гистограммы цен по 100+ категориям) с p95 < 100 мс.
- Бизнес-требование на семантический поиск (нативное в OpenSearch 2.x через k-NN).

## Альтернативы

- **Elasticsearch / OpenSearch** — отложен. Вернёмся при триггерах выше. **OpenSearch предпочтительнее ES** в продакшене из-за лицензии (Apache 2.0).
- **PostgreSQL FTS (`tsvector`)** — рассматривался как нулевая опция. Отклонён: нет typo-tolerance, нет фасетов с приемлемой производительностью.
- **Typesense** — близкий к Meilisearch конкурент. Отклонён: меньше зрелое JVM-SDK, менее активное русскоязычное сообщество.
